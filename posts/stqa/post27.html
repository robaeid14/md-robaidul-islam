Section 27: Testing on Object-Oriented Systems

1. Overview

Object-Oriented (OO) software systems introduce additional testing complexity due to features like inheritance, encapsulation, and polymorphism. OO testing requires both structural and behavioral validation of objects, classes, and their interactions.

2. Key Features Impacting Testing

OO Feature Testing Implication
Encapsulation Hard to access internal state; requires testing via public interfaces
Inheritance Changes in parent classes can affect child classes unintentionally
Polymorphism Method calls may resolve to multiple implementations at runtime
Dynamic Binding Runtime behavior may differ from static analysis expectations
Object States Each object can have unique states and transitions to test

3. Levels of Testing in OO Systems

  • Unit Testing: Testing individual classes and methods
  • Integration Testing: Testing interactions between classes (e.g., composition, inheritance chains)
  • System Testing: Testing complete object collaboration in workflows

4. Common OO Testing Techniques

a. Class Testing

  • Test each class in isolation, focusing on constructors, methods, and accessors.
  • Include tests for both valid and invalid states.

b. Inheritance-Based Testing

  • Ensure subclasses properly override methods and maintain Liskov Substitution Principle (LSP).
  • Test inherited behaviors and any new ones introduced by subclasses.

c. State-Based Testing

  • Model an object as a finite state machine (FSM).
  • Test all state transitions triggered by events or method calls.

d. Interaction Testing

  • Validate message passing and method invocation between objects.
  • Use mocks/stubs to isolate object dependencies.

5. Example

Class Hierarchy:
Animal → Dog, Cat

  • Test Animal's base methods (e.g., eat(), sleep())
  • Override speak() in Dog and Cat
  • Check if Dog.speak() returns "Bark" and Cat.speak() returns "Meow"

6. Tools for OO Testing

  • JUnit / NUnit: Popular for unit testing classes in Java/.NET
  • Mockito: For mocking dependencies in interaction tests
  • UML Tools: Helpful for modeling object states and transitions

7. Challenges

  • Code reuse through inheritance makes impact analysis harder
  • Polymorphism introduces multiple runtime paths to verify
  • Testing private methods and states can violate encapsulation

8. Best Practices

  • Design for testability – avoid excessive hidden states
  • Use mocking frameworks for loose coupling
  • Write tests for base and derived classes separately
  • Include regression tests when updating inherited code
